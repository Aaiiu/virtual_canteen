## 一、系统总体目标

### 1.1 项目背景

北京交通大学食堂是全校师生日常就餐的核心场所。每逢中午就餐高峰期（12:00-12:30），大量学生同时涌入食堂，导致各打饭窗口排队人数激增、等待时间显著延长，同时空闲座位难以快速定位，严重影响学生就餐体验和课间休息质量。

为解决上述问题，本项目设计并实现一款基于计算机仿真技术的虚拟食堂就餐仿真系统，通过对真实食堂场景的数字化建模与模拟，为学生提供数据支持和决策参考。

### 1.2 总体目标

1. **可视化呈现**：以2D平面图形式直观展示食堂实时状态，包括各菜系窗口排队情况与座位占用分布；
2. **排队仿真**：模拟学生到达食堂、选择窗口、排队等候、打饭就坐、就餐结束的完整流程；
3. **个体差异建模**：引入学生特性词条库，为不同学生随机分配行为特性，模拟真实人群个体行为差异对排队效率的影响；
4. **数据分析**：统计各时间段排队等待时间、人流量分布及座位占用率，标注高峰时段；
5. **智能推荐**：基于仿真数据推荐最佳就餐时间，并对比两种排队策略的效率差异；
6. **座位辅助**：实时显示空闲座位分布，辅助学生快速找到合适座位。

### 1.3 目标用户

- 北京交通大学在校学生（主要用户）
- 食堂管理人员（参考用户）
- 课程演示评审人员

### 1.4 系统边界

- 本系统为**仿真演示系统**，数据基于模拟生成，不与真实食堂设备对接；
- 仅需在本地环境运行，无需公网部署；
- 模拟范围为北京交通大学某一食堂的中午就餐时段（11:30-13:30）。

## 二、系统开发架构

### 2.1 架构类型：B/S架构（Browser/Server）

本系统采用 **B/S（浏览器/服务器）架构**，用户通过本地浏览器访问系统，无需安装任何客户端软件。

### 2.2 架构层次

    用户浏览器（Browser）
        HTTP请求 / JSON响应（AJAX）
    Flask Web服务器（Server）
        SQLAlchemy ORM
    SQLite 数据库

### 2.3 选择B/S架构的理由

| 对比维度 | B/S架构（本系统） | C/S架构 |
|----------|------------------|---------|
| 部署方式 | 无需安装，浏览器直接访问 | 需要安装客户端 |
| 跨平台性 | 支持Windows/macOS/Linux | 依赖客户端平台 |
| 演示便捷性 | 启动服务后即可演示 | 需分发安装包 |
| 开发分工 | 前后端分离，分工清晰 | 耦合度较高 |

### 2.4 通信方式

- 前端通过 **AJAX（Fetch API）** 向后端发起异步HTTP请求；
- 后端返回 **JSON格式** 数据；
- 前端定时轮询（每1-2秒一次）获取最新模拟状态，实现实时刷新效果。

## 四、系统架构

### 4.1 模块划分总览

本系统划分为五大模块：数据模型模块、仿真引擎模块、API接口模块、前端展示模块、数据存储模块。

    前端展示模块（2D可视化 | 悬停浮窗 | 图表 | 对比面板）
            |  HTTP/JSON
    API接口模块（Flask路由 | 数据序列化 | CORS）
         |                        |
    仿真引擎模块              数据模型模块
    排队模拟/特性词条   <-->   窗口/学生/座位/词条
    对比分析/时间分析              |
                            数据存储模块（SQLite）

### 4.2 各模块功能详述

#### 模块一：数据模型模块（src/models/）
负责人：成员A

| 子模块 | 文件 | 核心功能 |
|--------|------|----------|
| 窗口模型 | window.py | 打饭窗口编号、菜系名称、基础服务时间 |
| 学生模型 | student.py | 学生ID、姓名、菜品偏好、个体速度系数 |
| 特性词条模型 | trait.py | 词条名称、触发概率、效果类型和参数 |
| 座位模型 | seat.py | 座位编号、占用状态、关联学生ID、是否曾被使用 |
| 队列模型 | queue.py | 各窗口实时排队人数和预计等待时间 |
| 数据库配置 | database.py | 初始化SQLAlchemy连接，管理数据库会话 |

#### 模块二：仿真引擎模块（src/simulation/）
负责人：成员B

| 子模块 | 文件 | 核心功能 |
|--------|------|----------|
| 学生生成器 | student_generator.py | 按到达率生成学生，按概率分配特性词条 |
| 排队模拟器 | queue_simulator.py | 多窗口排队、偏好匹配、等待阈值切换 |
| 座位管理器 | seat_manager.py | 座位分配释放，孤独症学生专属逻辑 |
| 时间段分析 | time_analysis.py | 统计各时段等待时间，标注高峰期 |
| 对比模拟 | comparison.py | 策略A和策略B双路运行，输出对比数据 |
| 推荐引擎 | recommendation.py | 基于历史数据推荐最佳就餐时间 |

#### 模块三：API接口模块（src/routes/）
负责人：成员A

| 接口路径 | 方法 | 功能说明 |
|----------|------|----------|
| /api/status | GET | 获取食堂整体状态 |
| /api/windows | GET | 获取所有窗口排队信息 |
| /api/seats | GET | 获取所有座位状态 |
| /api/seat/{id} | GET | 获取指定座位学生详情（悬停浮窗用） |
| /api/simulate/start | POST | 启动新一轮模拟（传入超参数配置） |
| /api/simulate/status | GET | 获取当前模拟进度和实时数据 |
| /api/analysis/time | GET | 获取时间段分析数据 |
| /api/analysis/comparison | POST | 启动对比模拟，传入偏好参数 |
| /api/recommendation | GET | 获取最佳就餐时间推荐结果 |

所有接口统一返回JSON格式：{ "code": 200, "message": "success", "data": { ... } }

#### 模块四：前端展示模块（src/static/ + src/templates/）
负责人：成员C

| 子模块 | 文件 | 核心功能 |
|--------|------|----------|
| 主逻辑 | main.js | 定时轮询API，统筹各模块数据更新 |
| 2D可视化 | visualization.js | Canvas绘制食堂平面图、窗口、队列、座位 |
| 悬停浮窗 | tooltip.js | 监听鼠标事件，悬停座位时显示学生信息卡 |
| 数据图表 | charts.js | Chart.js渲染折线图、柱状图、饼图 |
| 对比面板 | comparison.js | 展示策略A vs 策略B的对比数据卡片 |

#### 模块五：数据存储模块

使用SQLite作为本地数据库，通过Flask-SQLAlchemy进行ORM操作，数据库文件位于 data/canteen.db。

### 4.3 模块间协同关系

    用户触发操作
        |
    前端展示模块 --AJAX请求--> API接口模块
                                    |
                               仿真引擎模块
                                |         |
                          数据模型模块  数据存储模块
                                |
                          计算结果返回
                                |
             API接口模块 --JSON--> 前端展示模块
                                        |
                               更新Canvas/图表/浮窗

### 4.4 模块间数据交换说明

| 数据交换方向 | 数据内容 | 接口形式 |
|-------------|----------|----------|
| 前端 -> API | 模拟启动参数（超参数、偏好设置） | HTTP POST / JSON Body |
| API -> 前端 | 窗口排队信息、座位状态、学生详情 | HTTP GET / JSON Response |
| API -> 仿真引擎 | 模拟配置参数 | Python函数调用 |
| 仿真引擎 -> 数据模型 | 学生对象、队列状态、座位分配结果 | SQLAlchemy ORM操作 |
| 数据模型 -> 数据存储 | 窗口/学生/座位/队列数据 | SQLite SQL语句 |
| 数据存储 -> API | 查询结果集 | SQLAlchemy查询对象 |

## 五、软件原型

### 5.1 界面总览

本系统包含三个主要页面，通过顶部导航栏自由跳转：

    主页（index.html）       --导航栏-->  分析页（analysis.html）
         |                                      |
         +----------导航栏----------->  对比页（comparison.html）

### 5.2 主页（index.html）

界面布局：

    顶部导航栏：[虚拟食堂仿真系统]  主页 | 数据分析 | 模拟对比
    左侧主区域（约70%宽）：
        Canvas画布绘制食堂2D平面图
        窗口区：[窗1-天津菜][窗2-湘菜][窗3-快餐][窗4-川菜]...
        座位区：200个座位格（绿色=空闲，红色=占用）
    右侧面板（约30%宽）：
        各窗口实时排队人数与预计等待时间列表
        当前在场总人数 / 座位空闲率
    底部控制栏：
        [启动模拟] [暂停] [重置]  模拟速度：[1x] [2x] [5x]

用户可进行的操作：
- 点击「启动模拟」开始仿真，点击「暂停」暂停，点击「重置」清空数据
- 拖动速度按钮调整模拟倍速（1x / 2x / 5x）
- 鼠标悬停已占用座位，弹出学生信息浮窗：

      +--------------------------+
      | 张三  学号: 21301001     |
      | 菜品偏好: 湘菜            |
      | 特性词条: 急性子           |
      | 就餐开始: 12:15           |
      | 剩余时间: 约3分钟          |
      +--------------------------+

- 点击导航栏跳转至分析页或对比页

### 5.3 数据分析页（analysis.html）

界面布局：

    顶部导航栏
    上半部分（左右各50%）：
        左：等待时间折线图（各时段各窗口对比，X轴=时间，Y轴=等待分钟）
        右：人流量柱状图（各时段到达人数，X轴=时间段，Y轴=人数）
    下半部分（左右各50%）：
        左：座位占用率饼图（高峰期/平峰期对比）
        右：特性人群分布图（各词条人群占比，环形图）
    底部推荐卡片：
        推荐时段：11:30-12:00  预计等待：约2分钟  座位充裕
        次选时段：12:45-13:00  预计等待：约4分钟  座位较充裕

用户可进行的操作：
- 鼠标悬停图表数据点，查看该时间点的详细数值
- 点击图例显示/隐藏特定窗口的折线
- 查看系统推荐的最佳就餐时间卡片
- 通过导航栏跳转

### 5.4 模拟对比页（comparison.html）

界面布局：

    顶部导航栏
    参数设置区：
        菜品偏好：[天津菜 v]  模拟人数：[500]  [开始对比]
    对比结果区（左右各50%并排）：
        策略A：按偏好就餐           策略B：选最短等待
        平均等待时间：8分钟         平均等待时间：4分钟
        平均寻座时间：2分钟         平均寻座时间：2分钟
        平均就餐时间：10分钟        平均就餐时间：10分钟
        平均总耗时：  20分钟        平均总耗时：  16分钟
    底部结论：
        策略B比策略A平均节省4分钟，但无法保证吃到偏好菜品，请根据个人情况权衡。

用户可进行的操作：
- 在下拉菜单中选择菜品偏好（对应各菜系窗口）
- 输入模拟人数（默认500人）
- 点击「开始对比」运行双策略仿真
- 查看两种策略在四个维度上的量化对比数据
- 查看系统自动生成的对比结论建议

### 5.5 界面跳转关系

三个页面均通过顶部导航栏自由跳转，无强制操作流程限制：

    主页 <--> 数据分析页 <--> 模拟对比页 <--> 主页

## 六、数据模型

### 6.1 系统数据总览

| 数据类别 | 产生位置 | 存储位置 | 流转方向 |
|----------|----------|----------|----------|
| 窗口配置数据 | 系统初始化脚本 | SQLite | 初始化 -> 数据库 -> API -> 前端 |
| 学生数据 | 仿真引擎（随机生成） | SQLite（临时） | 生成 -> 数据库 -> API -> 前端 |
| 特性词条数据 | config.py + 初始化脚本 | SQLite | 初始化 -> 数据库 -> 仿真引擎 |
| 排队队列数据 | 仿真引擎（实时更新） | 内存 + SQLite | 引擎 -> 数据库 -> API -> 前端 |
| 座位状态数据 | 仿真引擎（座位管理器） | SQLite | 引擎 -> 数据库 -> API -> 前端 |
| 模拟历史记录 | 仿真引擎（时间推进） | SQLite | 引擎 -> 分析模块 -> API -> 前端 |
| 对比分析结果 | 对比模拟模块 | 内存（临时） | 引擎 -> API -> 前端 |

### 6.2 数据库表结构

#### 表1：windows（窗口信息表）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | INTEGER PK | 窗口编号 |
| name | VARCHAR(50) | 窗口名称 |

#### 表2：students（学生信息表）

| 字段名 | 类型 | 说明 |

| id | INTEGER PK | 学生ID |

#### 表3：traits（特性词条库表）

| 字段名 | 类型 | 说明 |

| 字段名 | 类型 | 说明 |
|--------|------|------|

### 6.3 数据流转图

    系统初始化
      init_database.py 写入窗口配置、特性词条库到数据库
              |
    仿真启动（用户点击「启动模拟」）
              |
      student_generator.py
        按到达率生成学生对象
        按概率分配特性词条
        写入 students 表、student_traits 表
              |
      queue_simulator.py
        计算各窗口排队状态（考虑偏好、特性词条影响）
        更新 queues 表
        写入 simulation_logs 表（每分钟一条记录）
              |
      seat_manager.py
        为完成打饭的学生分配座位
        就餐结束后释放座位
        更新 seats 表
              |
    API接口读取数据库 -> 返回JSON给前端
              |
    前端实时更新（每1-2秒轮询一次）
      visualization.js 更新Canvas画布（队列长度、座位颜色）
      charts.js 更新统计图表
      tooltip.js 更新悬停浮窗内容

### 6.4 数据生命周期

| 数据类型 | 生命周期 | 说明 |
|----------|----------|------|
| 窗口配置 | 持久化 | 初始化后长期存储，不随模拟重置 |
| 特性词条库 | 持久化 | 系统级配置，可通过config.py调整概率后重新初始化 |
| 学生数据 | 单次模拟 | 每次重置后清空并重新生成 |
| 队列数据 | 实时更新 | 随仿真时间步进持续更新，历史记录写入simulation_logs |
