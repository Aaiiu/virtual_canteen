# 虚拟食堂仿真系统 - 项目规划文档

## 📋 项目概述

### 项目目标
开发一个食堂排队模拟系统，帮助学生：
- 预测不同时间段的排队等待时间
- 可视化展示食堂实时状态
- 智能推荐最佳就餐时间
- 辅助快速找到空闲座位

### 团队信息
- **团队规模**：3人
- **技术背景**：C++基础，部分Java基础
- **开发周期**：根据课程安排（具体见时间规划）

---

## 🎯 核心功能模块

### 优先级划分

#### P0 - 核心必须功能（第一阶段）
1. **食堂2D可视化界面**
   - 显示食堂布局（窗口、座位区域）
   - 实时显示排队人数
   - 座位占用情况展示

2. **排队模拟系统**
   - 模拟不同时间段的人流量
   - 计算预估等待时间
   - 多窗口排队逻辑

3. **基础数据展示**
   - 各窗口当前排队人数
   - 预计等待时间
   - 座位空闲率

#### P1 - 重要功能（第二阶段）
4. **时间段分析**
   - 历史数据统计
   - 高峰时段标注
   - 最佳就餐时间推荐

5. **座位查找辅助**
   - 空闲座位实时显示
   - 座位区域推荐

#### P2 - 可选功能（时间充裕时）
6. **3D可视化升级**
7. **用户偏好设置**
8. **数据导出功能**

## 💻 技术架构方案

### 推荐技术栈（适合初学者）

#### 方案A：Python + Web（推荐⭐⭐⭐⭐⭐）
**适合理由**：语法简单，学习曲线平缓，库丰富

**前端**：
- HTML + CSS + JavaScript（原生，无框架）
- Bootstrap（UI组件库，开箱即用）
- Chart.js（图表展示）
- Canvas API（2D可视化）

**后端**：
- Python 3.x
- Flask（轻量级Web框架，易学）
- SQLite（无需安装配置的数据库）

**模拟引擎**：
- SimPy（Python离散事件模拟库）
- 或自己实现简单的队列模拟逻辑

**优势**：
- Python语法接近自然语言，容易上手
- Flask框架简单，几行代码就能跑起来
- SQLite无需配置，直接使用
- 大量中文教程和文档

---

#### 方案B：Java + Web（备选）
**适合理由**：部分成员有Java基础

**前端**：同方案A

**后端**：
- Java 8+
- Spring Boot（简化配置）
- H2 Database（内存数据库）

**优势**：
- 利用现有Java知识
- Spring Boot自动配置，降低难度

**劣势**：
- 配置相对复杂
- 学习曲线较陡

---

### 推荐选择：方案A（Python + Flask）

## 🔄 Git工作流方案

### 推荐：GitHub + GitHub Flow（简化版）

#### 为什么选择GitHub？
1. **免费且功能强大**：私有仓库免费，适合小团队
2. **界面友好**：Web界面直观，易于学习
3. **文档丰富**：大量中文教程
4. **集成工具多**：Issues、Projects、Wiki等

#### 分支策略

```
main (主分支)
  ├── dev (开发分支)
  │    ├── feature/queue-simulation (功能分支-排队模拟)
  │    ├── feature/visualization (功能分支-可视化)
  │    └── feature/data-analysis (功能分支-数据分析)
```

**分支说明**：
- `main`：稳定版本，只接受来自dev的合并
- `dev`：开发主分支，日常开发在此进行
- `feature/*`：功能分支，开发具体功能

#### 工作流程

1. **克隆仓库**
```bash
git clone https://github.com/your-team/Virtual_Canteen.git
cd Virtual_Canteen
```

2. **创建功能分支**
```bash
git checkout dev
git pull origin dev
git checkout -b feature/your-feature-name
```

3. **开发并提交**
```bash
git add .
git commit -m "feat: 添加排队模拟功能"
git push origin feature/your-feature-name
```

4. **创建Pull Request**
- 在GitHub网页上创建PR
- 请求团队成员Review
- 通过后合并到dev分支

5. **定期合并到main**
- 每完成一个阶段性功能
- 经过测试无误后合并

#### Commit规范

使用简单的提交信息格式：
```
feat: 新增功能
fix: 修复bug
docs: 文档更新
style: 代码格式调整
refactor: 代码重构
test: 测试相关
```

示例：
```bash
git commit -m "feat: 实现排队模拟基础逻辑"
git commit -m "fix: 修复窗口人数计算错误"
git commit -m "docs: 更新API文档"
```

## 附录：常见问题FAQ

### Q1: 我们完全没有Web开发经验，能完成吗？
A: 可以的。Python和Flask都是对初学者友好的技术，跟着教程一步步来，2周内可以掌握基础。项目规划已经考虑了学习时间。

### Q2: GitHub用不习惯怎么办？
A: 可以使用GitHub Desktop图形界面工具，比命令行更直观。也可以使用VS Code的Git集成功能。

### Q3: 排队模拟算法很复杂吗？
A: 不复杂。可以使用简单的队列数据结构，配合时间推进逻辑即可。也可以使用SimPy库简化开发。

### Q4: 2D可视化怎么实现？
A: 使用HTML Canvas API，就像画图一样。可以参考在线教程，有很多现成的例子。

### Q5: 如果进度落后怎么办？
A: 优先保证P0功能完成，P1和P2可以根据实际情况调整。最重要的是有一个能演示的系统。

### Q6: 需要学习数据库吗？
A: SQLite非常简单，基本的增删改查操作1-2天就能学会。而且Flask-SQLAlchemy会进一步简化操作。

### Q7: 前端需要学习框架吗？
A: 不需要。使用原生HTML/CSS/JavaScript配合Bootstrap就足够了。学习框架反而增加负担。

### Q8: 代码冲突怎么解决？
A: 按照Git工作流，每人在自己的分支开发，定期合并。如果有冲突，团队一起解决。前期做好模块划分可以减少冲突。
